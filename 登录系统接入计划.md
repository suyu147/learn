### 三个断裂点

1. **login page 自己写 fetch + `document.cookie`**，而 api-client 从 `auth-token.ts`（模块变量）读 token → 登录后 api-client 拿不到 token，后续请求无 Authorization 头
2. **`initAuth()` POST 空密码探测 mode** → 语义错误，且后端 login 路由对 disabled 模式返回假 token，不返回 mode 信息
3. **根 layout 硬编码 `<Sidebar />`** → auth 页面也显示侧边栏

## 1. 改动清单

总共 **3 个修改 + 1 个新建 = 4 个文件**。

### 1.1 修复 `initAuth()` — 1 个文件

**文件**：`lib/store/auth-store.ts`

**改动**：`initAuth()` 从 POST `/api/v1/auth/login`（空密码）改为 GET `/api/v1/auth/status`

```
改前：POST /api/v1/auth/login { username: '', password: '' } → 猜 mode
改后：GET  /api/v1/auth/status                    → { mode, user }
```

**逻辑**：

- `mode === 'disabled'`：直接拿到 local-admin user，无需登录
- `mode === 'single'`：直接拿到自动创建的单用户，无需登录
- `mode === 'multi'`：user 为 null，前端需显示登录页

**注意**：`/api/v1/auth/status` 在 disabled/single 模式下会调用 `authenticate()` 返回用户信息，但**不会返回 JWT token**。需要在 status 路由中为 disabled/single 模式补充签发 token（或直接调用 `createToken`），否则前端无法拿到有效 token 供 api-client 使用。

### 1.2 修复 login / register 页面 — 2 个文件

**文件**：
- `app/auth/login/page.tsx`
- `app/auth/register/page.tsx`

**改动**：`handleSubmit` 中 `fetch + document.cookie` 替换为 `useAuthStore.login()` / `useAuthStore.register()`

```
改前：
  const res = await fetch('/api/v1/auth/login', { ... });
  document.cookie = `auth_token=${data.data.token}; ...`;
  router.push('/chat');

改后：
  const login = useAuthStore(s => s.login);
  await login(username, password);  // 内部 setApiToken()
  router.push('/chat');             // api-client 已能读到 token
```

UI 和样式完全不动。

### 1.3 路由守卫 + Sidebar 隔离 — 1 个新建 + 1 个修改

**问题**：multi mode 下未登录用户直接访问 `/chat` 会看到空页面；auth 页面显示了 Sidebar。

**方案**：不拆路由组（改动太大），改用条件渲染。

#### 新建 `components/auth-guard.tsx`

```typescript
'use client';
// 在 Providers 内部，根据 auth state 决定：
//   - multi + 未登录 + 非 /auth/* → 重定向 /auth/login
//   - multi + 已登录 + /auth/* → 重定向 /chat
//   - 其他 → 正常渲染
```

#### 修改 `app/layout.tsx`

```
改前：
  <Providers>
    <div className="flex h-screen">
      <Sidebar />
      <main>{children}</main>
    </div>
  </Providers>

改后：
  <Providers>
    <AppShell>{children}</AppShell>   ← Sidebar 移入 AppShell，由 auth state 控制
  </Providers>
```

`AppShell` 逻辑：
- 如果 `mode === 'multi' && !user && pathname 不以 /auth/ 开头` → 重定向登录
- 如果 pathname 以 `/auth/` 开头 → 渲染居中布局（无 Sidebar）
- 否则 → `<Sidebar /> + <main>{children}</main>`

**注意**：`AppShell` 可以直接写在 `components/providers.tsx` 里，不必新建文件。这样只改 `app/layout.tsx` 一个文件。

### 1.4 补充 `/api/v1/auth/status` 的 token 签发

**文件**：`app/api/v1/auth/status/route.ts`

**改动**：disabled / single 模式下，除了返回 user 信息，还签发 JWT token 一并返回。

```
改前：
  { mode, user: { id, username, role } }

改后：
  { mode, user: { id, username, role }, token }   ← 仅 disabled/single 有
```

这样 `initAuth()` 拿到 token 后调 `setApiToken()`，后续请求自动带 Authorization。

## 4. Sidebar 用户信息接入

当前 Sidebar 底部硬编码了"陈思远"（`components/sidebar.tsx:179-186`）。

登录系统完成后，改为从 `useAuthStore` 读取：

```
改前：
  <div>陈思远</div>
  <div>学习者 · 大三</div>

改后：
  const user = useAuthStore(s => s.user);
  const logout = useAuthStore(s => s.logout);
  <div>{user?.username ?? 'Guest'}</div>
  <div>{user?.role === 'admin' ? '管理员' : '用户'}</div>
  + 退出按钮 onClick={logout}
```

这一步改动极小（~5 行），可以和 Step 5 一起做。
