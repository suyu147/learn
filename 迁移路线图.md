# DeepTutor → Next.js 全栈迁移路线图

> 基于 SmartLearn 基底，将 DeepTutor 全部功能迁移为 Next.js 全栈架构

## 项目位置

- DeepTutor 源码：`E:\DeepTutor-main`
- SmartLearn 源码：`D:\python\docment\smartlearn`

## 总体原则

- **每个 Phase 结束后项目可运行、可验证**
- **先搭骨架再填肉**：架构模式 → 基础设施 → 核心功能 → 扩展功能
- **SmartLearn 已有的不重写**：LLM Provider、SSE 流式、LangGraph 编排、资源查看器等直接复用
- **不实现 CLI 系统**
- **Partner/Cron/Subagent 推迟到后期**（需要长连接/持久进程，与 Next.js 不兼容）
- **仅自部署模式**，不支持 serverless 部署

---

## 关键架构决策

### 决策 1：Agentic Loop 基于 LangGraph 实现

所有 Capability 都是 LangGraph 子图，通过 Orchestrator 路由。三种类型：

| 类型 | 实现 | 适用场景 |
|---|---|---|
| **LoopCapability** | AgentLoop 子图（LLM → Tool → LLM 循环） | chat/solve/mastery/explore_context/obsidian |
| **AgentCapability** | 线性多阶段子图（阶段 A → B → C） | deep_question/deep_research/visualize/math_animator/vision_solver/notebook |
| **GraphCapability** | 自定义状态机子图（条件边 + 闭环） | smartlearn（learning-graph 有循环：evaluate → plan → generate） |

### 决策 2：SSE + HTTP POST 回传（仅自部署 + 超时降级）

- **正向流**：SSE（与 SmartLearn 现有模式一致）
- **ask_user 回传**：SSE 发 `WAIT_FOR_INPUT` → 前端 POST `/api/v1/input` → 服务端 Promise 恢复
- **跨 handler 状态共享**：单 worker 模式 + 模块级 `Map<turnId, PromiseResolver>`，SSE handler 写入 Map，POST handler 读取并 resolve
- **连接保持**：ask_user 暂停期间 SSE 连接保持，服务端 `await Promise` 等待用户输入（自部署模式下连接数不是瓶颈）
- **超时降级**：ask_user 超时（默认 60s）后自动转为普通文本回复，释放连接
- **仅自部署模式**，不支持 Vercel serverless

### 决策 3：Prisma + PostgreSQL

- 复用 SmartLearn 已有的 Prisma + PostgreSQL
- 扩展 `schema.prisma` 加入 Session/Turn/Message 表
- TurnEvent 仅持久化关键事件（RESULT/TOOL_CALL/TOOL_RESULT），其余丢弃
- Message.events 用文件存储替代 Json 列（避免单条消息数百 KB）

### 决策 4：RAG 分阶段实现

- Phase 2b 仅基础向量检索（pgvector + 文档分块 + 单向量检索 + 基本重排序）
- BM25 混合、4 引擎、索引版本化推迟到 Phase 5
- 不依赖 `llamaindex-ts`，用 pgvector + 自研分块/检索

### 决策 5：Parsing 仅 markitdown + 纯文本

- 不引入 Python sidecar，MinerU/Docling 不实现

### 决策 6：Sandbox 复用 SmartLearn 的 Piston API

- 不自建 subprocess/runner/bwrap 后端

### 决策 7：代码放在 `lib/deeptutor/` 下

- 与 SmartLearn 的 `lib/ai/`、`lib/store/` 同级
- import 路径统一：`@/lib/deeptutor/...`

### 决策 8：SmartLearn 改造为 GraphCapability

- learning-graph 保留为 smartlearn GraphCapability 的内部实现（带条件边和闭环）
- director-graph 保留为 smartlearn 内部的 PPT 编排（**不复用为 Chat Capability**）
- Chat Capability 用新的 AgentLoop 子图
- PPT/Stage/Canvas/7 种资源查看器保留为 smartlearn 专属组件

### 决策 9：废弃 SmartLearn 旧 API 路由，统一 `/api/v1/*`

- SmartLearn 的 `/api/learn`、`/api/chat` 等 16 个旧路由全部废弃
- 旧路由的 handler 逻辑迁移到新路由对应的 Service 层
- **过渡策略**：Phase 2d 统一切换，SmartLearn 前端一次性迁移到新 API，旧 API 同步废弃

### 决策 10：Partner/Cron/Subagent 全部推迟

- 这些功能需要长连接/持久进程/子进程，与 Next.js 请求-响应模型不兼容
- 推迟到项目稳定后再设计独立进程方案

### 决策 11：单 worker 部署模式

- `next start` 限制为 1 个 worker，用模块级变量共享状态
- ask_user 的 `Map<turnId, PromiseResolver>`、EventBus、Turn 取消等均依赖此模式
- 后期如需多 worker，需引入 Redis pub/sub 替代模块级状态

### 决策 12：EventBus 短期用模块级 EventEmitter

- 短期：模块级 EventEmitter（仅单 worker 可用，满足当前需求）
- 长期：如需多 worker 或跨进程通信，升级为 Redis pub/sub
- 用途：CAPABILITY_COMPLETE 触发记忆整合等跨模块通信

### 决策 13：KnowledgeCapability（独占工具面）

- LoopCapability 增强工具面（追加工具），KnowledgeCapability 替换整个工具面
- Obsidian/Subagent 是 KnowledgeCapability：选中 Obsidian vault 或 Subagent KB 时，工具面从通用切换为专属工具集
- Phase 0 必须定义 `exclusiveTools` 标记和替换逻辑

## 设计规范

### Prisma Session/Turn/Message Schema

```prisma
model Session {
  id        String   @id @default(cuid())
  userId    String?
  mode      String   // chat | smartlearn | solve | mastery | book | co_writer
  title     String?
  metadata  Json?
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
  turns     Turn[]
  messages  Message[]
}

model Turn {
  id        String   @id @default(cuid())
  sessionId String
  userId    String?
  status    String   @default("running") // running | completed | failed | cancelled
  mode      String?  // chat | smartlearn | solve | ...
  metadata  Json?
  createdAt DateTime @default(now())
  session   Session  @relation(fields: [sessionId], references: [id], onDelete: Cascade)
  messages  Message[]
}

model Message {
  id              String   @id @default(cuid())
  sessionId       String
  turnId          String?
  role            String   // user | assistant | system | tool
  content         String?
  parentMessageId String?  // Branch: 指向父消息，实现分支
  metadata        Json?
  createdAt       DateTime @default(now())
  session         Session  @relation(fields: [sessionId], references: [id], onDelete: Cascade)
  turn            Turn?    @relation(fields: [turnId], references: [id], onDelete: Cascade)
}
```

**TurnEvent 关键事件**：RESULT, TOOL_CALL, TOOL_RESULT（仅这 3 种持久化，其余丢弃）

**Message.events 文件存储**：`data/user/{userId}/events/{messageId}.json`，访问 API `GET /api/v1/messages/:id/events`，随 Message 删除而清理

### 认证与配置过渡

- **Phase 1 直接实现三种认证模式**（无认证/单用户/多用户），默认单用户模式
- SmartLearn 的 `DEFAULT_USER_ID` 改为从认证上下文获取（`AsyncLocalStorage`）
- **API Key 统一迁移到服务端**：Phase 1 建服务端 settings API，API Key 存 PostgreSQL（加密），SmartLearn 客户端 localStorage Key 废弃
- 模型目录（model_catalog）统一为服务端 JSON，前端只读展示
- 设置变更通过 `EventBus` 通知 LangGraph 节点（配置热更新）

### 运行时数据目录

```
data/
  user/
    {userId}/
      memory/          ← L1/L2/L3 记忆文件
      notebooks/       ← 笔记本数据
      skills/          ← 技能包
      sandbox/         ← 沙箱制品
      book-output/     ← Book 生成输出
      events/          ← Message.events 文件存储
      settings/        ← 用户级 JSON 设置
```

- 多用户隔离：按 `{userId}` 分目录
- 访问方式：通过 Storage 服务抽象，支持本地磁盘 + S3/MinIO/GCS
- Next.js 通过 `process.env.DATA_DIR` 配置根目录

### Prompt 块顺序（ChatPromptAssembler）

实际顺序（来自 DeepTutor 源码 `prompt_blocks.py`）：

1. general — 产品身份 / Partner 身份
2. runtime_policy — 运行时策略
3. loop — AgentLoop 循环指令
4. capability_blocks — 活跃 Capability 的 Prompt 块（可多个）
5. persona_style — 人格风格
6. partner_turn_policy — Partner 轮次策略
7. memory — 记忆上下文
8. tools — 工具清单（含 kb_note 前缀）
9. skills — 技能包清单
10. sources — 附加源清单
11. extended_tools — 延迟工具 manifest（DeferredToolLoader）
12. notebooks — 笔记本 manifest
13. workspace — 工作区备注

**KB Seed**：不注入 system prompt，追加到 user_message 末尾（保持 system prompt 字节稳定）

### SmartLearn 事件映射（StatelessEvent → StreamEvent）

| SmartLearn StatelessEvent | DeepTutor StreamEvent | 说明 |
|---|---|---|
| text_delta / text-delta | CONTENT | 文本流式输出 |
| thinking | THINKING | 思考/推理过程 |
| tool-call | TOOL_CALL | 工具调用 |
| tool-result | TOOL_RESULT | 工具结果 |
| agent_start | STAGE_START | Agent/阶段开始 |
| agent_end | STAGE_END | Agent/阶段结束 |
| agent-switch | STAGE_START | Agent 切换（映射为阶段开始） |
| action | TOOL_CALL | 白板/聚光灯等动作（映射为工具调用） |
| cue_user | WAIT_FOR_INPUT | 提示用户输入 |
| done | DONE | 流结束 |
| error | ERROR | 错误 |

**SmartLearn 无对应的事件**：OBSERVATION, PROGRESS, SOURCES, RESULT, SESSION, SESSION_META（DeepTutor 特有，SmartLearn 不需要）

### LLM 服务适配

- **UsageTracker**：LangGraph 节点内调用 `streamLLM` 后，从 AI SDK response 提取 `usage` 字段，累加到 `TurnState.tokenUsage`
- **TrafficController**：模块级信号量（RPM 限流 + 并发上限），包装 `streamLLM` 调用
- **子 Agent LLM 模式**：research/question 的子 Agent 有独立 temperature/max_tokens，通过 `CapabilityManifest.configDefaults` 覆盖
- **能力检测**：`supportsTools(modelId)` / `supportsVision(modelId)` 查询模型目录的能力标记

## LangGraph State 与 UnifiedContext 映射（Phase 0 必须定义）

| 概念 | DeepTutor Python | Next.js TS |
|---|---|---|
| 请求上下文 | `UnifiedContext` dataclass | `UnifiedContext` interface，通过 LangGraph `config.configurable` 传入 |
| 图状态 | `Annotation.Root` + reducer | `Annotation.Root` + reducer（LangGraph JS 原生） |
| 累加字段 | `generatedResources: Annotation(list, reducer=add)` | `generatedResources: Annotation<string[]>({ reducer: (_, old, update) => [...old, ...update] })` |
| 消息历史 | `messages: Annotation(list, reducer=add_messages)` | `messages: Annotation<BaseMessage[]>({ reducer: messagesReducer })` |
| 私有 kwargs | `_sandbox_*` 等，通过 `**kwargs` 注入 | 通过 `config.configurable` 传入，tool_node 执行时读取 |

**调用方式**：`graph.invoke({ messages, ...stateFields }, { configurable: { context, sessionId, userId } })`

## Prompt 管理系统（Phase 0 设计）

DeepTutor 的 PromptManager（YAML + Jinja2 + i18n）迁移为 TS 版：

```
lib/deeptutor/core/prompt/
  manager.ts            ← PromptManager：加载 YAML + 渲染 + 缓存 + i18n 回退
  renderer.ts           ← 模板渲染（用 Handlebars 替代 Jinja2，{{variable}} 语法）
  types.ts              ← PromptBlock, PromptTemplate 类型
```

- YAML 模板用 `js-yaml` 加载
- 模板渲染用 Handlebars（`{{variable}}` 语法，与 Jinja2 的 `{{variable}}` 兼容）
- i18n 回退：zh → en → 默认值
- 运行时变量通过 `render(templateId, variables)` 注入
